# § 3 API運用時の留意事項

出典: デジタル庁「APIテクニカルガイドブック」§ 3

---

## 1. 利用者への告知

APIの変更・廃止・障害等の情報は、API利用者に事前かつ適切に告知すること。

| 告知内容 | 留意点 |
|---------|--------|
| ベースURIの変更 | 運用開始後は原則変更しない。変更する場合は前もって開発者へ通知 [SHOULD] |
| バージョンアップ・廃止 | API利用者が移行できる十分な期間を設けて告知 |
| 障害・メンテナンス | 503 Service Unavailableを適切に使用し、メンテナンス情報を告知 |

---

## 2. モニタリングとロギング

### モニタリング
- APIのパフォーマンス・可用性を継続的に監視する [SHOULD]
- 監視すべき主な指標:

| 指標 | 概要 |
|------|------|
| レスポンスタイム | `X-Response-Time` ヘッダーで返却するレスポンス時間 |
| エラーレート | 4xx / 5xx 系ステータスコードの発生率 |
| リクエスト数 | 単位時間あたりのリクエスト数（負荷状況の把握） |
| 可用性 | サービスの稼働率 |

### ロギング
- リクエスト・レスポンスを記録し、障害調査・利用状況分析に活用する [SHOULD]
- `X-Request-ID` ヘッダーにより、ログのトレーサビリティを確保する
- ログに個人情報・機密情報が含まれないよう注意すること

---

## 3. SLAとSLOの設定

### SLA（Service Level Agreement）
- APIサービスの品質保証レベルをAPI利用者と合意した取り決め
- 主な記載事項:

| 項目 | 内容例 |
|------|--------|
| 可用性 | 99.9% uptime |
| レスポンスタイム | 平均500ms以内 |
| 障害時の対応時間 | 2時間以内に一次対応 |
| メンテナンスウィンドウ | 毎週日曜 2:00〜4:00 |

### SLO（Service Level Objective）
- SLAを達成するための内部目標値（SLAより厳しく設定することが多い）
- SLOを継続的に計測・モニタリングすることで、SLA違反を未然に防ぐ

### 設定のポイント [SHOULD]
- SLA・SLOはAPI提供開始前に定義し、ドキュメントに明記する
- 定期的にSLOの達成状況を評価し、必要に応じてインフラや実装を改善する
