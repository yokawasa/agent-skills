# § 2.3 個別データの各パラメータについて

出典: デジタル庁「APIテクニカルガイドブック」§ 2.3

---

## 基本方針

データの利活用には他サービスとの結合が重要なため、国際標準・デファクト標準に従ってデータ項目名と内容を設定することを推奨する [SHOULD]。

---

## 推奨する国際標準表記

| 内容 | 標準 | 表記例 |
|------|------|--------|
| 日時（日付のみ） | ISO 8601 | `2017-02-06` |
| 日時（日本時間） | ISO 8601 | `2017-02-06T13:50:40+0900` |
| 言語 | ISO 639-1（多言語対応時はISO 639-2/3も検討） | `ja`（日本語）, `en`（英語） |
| 貨幣 | ISO 4217 | `JPY`（日本円）, `USD`（米ドル） |
| 性別 | ISO 5218 | `1`（男性）, `2`（女性）, `9`（その他）, `0`（不明） |
| 国 | ISO 3166-1 alpha-2 | `JP`（日本）, `US`（米国） |
| 都道府県 | JIS X 0401 | `13`（東京都） |
| 市区町村 | JIS X 0402 | `101`（東京都千代田区） |
| 公衆警報の深刻度カラーコード | ISO 22324 | 赤（危険）, 黄（注意）, 緑（安全） |
| 地理座標 | GeoJSON（RFC 7946）、測地系はJGD 2011/(B,L) | `35.676097, 139.744879` |

> 日本語は必須、英語は推奨とする（ISO 639-1）

> 全項目を一律に用意する必要はなく、APIの仕様に合致する必要な項目のみを選択 [SHOULD]

> 標準準拠項目と追加項目が区別できるようにAPI仕様書で明示すること [SHOULD]

---

## データ項目設計指針

### ① データ項目名 [SHOULD]
- **分かりやすい名詞・複数形**を使用
- **スネイクケース**推奨: `city_libraries`
- 曖昧な名称や不適切な英語名は、他サービス連携時のデータ不正・不備の原因になるため禁止

### ② 入力規約
- 表現・記述方法が複数存在するもの（日時等）や記述ルールが必要なもの（法人名ヨミガナ等）はAPI仕様書内で**入力規約**として明文化する

### ③ 個別記述 vs 列挙
設計上の判断基準:

| 記述方法 | 適する場合 |
|---------|-----------|
| **個別記述**（各曜日ごとに項目） | 各要素の「属性（○/×等）」を示す場合 |
| **列挙**（条件に合致するものをカンマ区切り等で列記） | 条件に合致する要素が「何か」を示す場合 |

> その他、変更への柔軟性・項目数等を勘案して決定する

### ④ コード（分類体系）
- 各種標準コードが存在する場合は必ず調査して活用する [SHOULD]
- 独自コードを設計する場合は「コード（分類体系）設計・運用実践ガイドブック」を参照

### ⑤ 構造化
- データの構造化・共通化により、API利用者の利便性と開発効率を高める
- 詳細は「マスターデータ等基本データ導入実践ガイドブック」を参照
