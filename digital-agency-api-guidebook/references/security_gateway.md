# § 2.4 セキュリティ・APIゲートウェイ・省庁間API標準

出典: デジタル庁「APIテクニカルガイドブック」§ 2.4

---

## 1. REST APIのセキュリティ対策

### セキュリティ対策一覧

| 観点 | 対策事項 | 規範強度 |
|------|---------|---------|
| 通信の改ざん・盗聴防止 | TLS 1.3（HTTPS通信）による暗号化 | **[MUST]** |
| HTTPS強制 | HSTS（Strict-Transport-Security）ヘッダーの実装 | **[MUST]** |
| 証明書管理 | 信頼できる認証局から取得し適切に管理 | **[MUST]** |
| エンドツーエンド暗号化 | 機密データは転送中・保存時ともに暗号化 | **[SHOULD]** |
| API利用者認証 | APIキーまたはOpenID ConnectによるAPI利用者認証 | **[SHOULD]** |
| 情報漏えい防止 | APIキーをパラメータに含めない、不要なAPIキーは削除 | **[SHOULD]** |

### 認証レベルの使い分け

| API種別 | 推奨認証方式 |
|---------|------------|
| 一般的なWebAPI | APIキー（最低限）以上の認証レベル |
| 行政手続API・個人情報を扱うAPI | **OpenID Connect** を推奨 [SHOULD] |
| 極めて守秘性の高い情報 | OpenID Connect **Hybrid Flow** を採用 [SHOULD] |

> **注意**: APIキーはコピーされてアクセスされるリスクがあるため、強固な対策ではない。認証レベルを必ずAPIの要件に応じて適切に選定すること。

> **OpenID Connect認定実装**: http://openid.net/certification/

## 2. API呼び出し先と呼び出し元の処理における整合性担保

- API呼び出し先（サーバー）と呼び出し元（クライアント）の処理が整合するように設計する
- 特にトランザクション管理・エラー時のロールバック等を考慮すること

---

## 3. APIゲートウェイの活用

### APIゲートウェイとは

API呼び出しを一元管理する中間レイヤー。以下の機能を提供する。

| 機能 | 内容 |
|------|------|
| 認証・認可 | APIキーやOAuthトークンの検証を集中管理 |
| レート制限 | 特定のクライアントからの過剰なリクエストを制限 |
| ロギング・モニタリング | リクエスト・レスポンスのログ収集と分析 |
| キャッシング | 頻繁なリクエストへのレスポンスをキャッシュしてパフォーマンス向上 |
| SSL/TLS終端 | HTTPS通信の暗号化・復号を集中処理 |
| リクエスト変換 | クライアントとバックエンドAPI間でのデータ形式変換 |
| 負荷分散 | 複数のバックエンドサーバーへのリクエスト分散 |

### 活用メリット
- セキュリティの強化と一元管理
- 開発・運用の効率化
- APIの変更・バージョン管理の容易化

## 4. 省庁間データ交換のためのAPI標準規格

省庁間でデータを交換する際は、以下の標準規格の活用を検討すること。

| 規格名 | 概要 |
|--------|------|
| OpenAPI Specification | REST APIの記述・共有のための標準仕様 |
| JSON Schema | JSONデータの構造・バリデーションの記述 |
| OAuth 2.0 / OpenID Connect | 認可・認証の標準プロトコル |
| HAL（Hypertext Application Language） | リソース間のハイパーリンク表現 |

> 省庁間APIの具体的な標準規格については、政府相互運用性フレームワーク（GIF）関連文書を参照すること。
