# デジタル庁 APIテクニカルガイドブック スキル

**出典**: デジタル社会推進実践ガイドブック DS-464-2「APIテクニカルガイドブック」  
（デジタル庁, 2024年9月30日）  
**位置づけ**: 政府相互運用性フレームワーク（GIF）DS-400 の一部

---

## このスキルの使い方

1. **設計支援**: 「〜なAPIを設計したい」→ ガイドブック準拠の仕様・URI・レスポンス構造を生成
2. **設計レビュー**: 「この設計を政府標準でレビューして」→ 各項目をチェックし、指摘と改善案を提示
3. **仕様確認**: 「〜の正しい書き方は？」→ 該当するガイドライン箇所を提示

---

## 規範強度の表記ルール

ガイドブックの表現を以下の3段階で統一して伝えること：

| 記号 | ガイドブックの表現 | 意味 |
|------|-------------------|------|
| **[MUST]** | 「〜こととする」「必須」 | 必ず従うべき |
| **[SHOULD]** | 「〜を推奨する」「推奨」 | 基本的に従うべき、合理的理由があれば外れてよい |
| **[MAY]** | 「〜することができる」「任意」 | 選択的に採用 |

## クイックリファレンス（参照ファイル不要な頻出事項）

### URI設計の基本原則 [SHOULD]
- 名詞・複数形を使用（動詞は不可）
- スネイクケース（`city_libraries`）、キャメルケース非推奨
- バージョン番号を含める: `v1`, `v2`（小数禁止）
- `api` サブドメインを推奨: `https://api.example.go.jp/v1/`
- 操作はHTTPメソッドで表現（URIに動詞を入れない）

### HTTPメソッドのCRUD対応 [SHOULD]
| 操作 | メソッド |
|------|---------|
| Create | POST |
| Read | GET |
| Update（置換） | PUT |
| Update（部分更新） | PATCH |
| Delete | DELETE |

> ⚠️ このマッピングは **[SHOULD]**（推奨）であり、**[MUST]** ではない。回答・仕様生成時に `[MUST]` と表記しないこと。

### 同期 vs 非同期の選択基準
| 条件 | 選択 |
|------|------|
| 処理時間 < 1秒、即時結果が必要 | 同期API（200 OK） |
| 処理時間 ≥ 1秒、バックグラウンド処理可 | 非同期API（202 Accepted） |

### レスポンス形式 [SHOULD]
- **JSON推奨**（軽量・普及度高）
- 文字コード: **UTF-8** [MUST]
- 外字使用禁止 [MUST]

## レビュー時のチェックリスト

API仕様のレビューを求められた場合、以下の観点で確認すること：

### URI設計
- [ ] 名詞・複数形になっているか
- [ ] スネイクケースを使用しているか
- [ ] バージョン番号（v1等）がURIに含まれているか
- [ ] `api` サブドメインを使用しているか
- [ ] HTTPメソッドで操作を表現しているか（URIに動詞がないか）

### レスポンス
- [ ] 適切なHTTPステータスコードを返しているか（references/response.md参照）
- [ ] 正常レスポンスにメタデータ（metadata, resultset等）を含んでいるか
- [ ] エラーレスポンスがRFC 7807形式（status/type/title/detail/instance）になっているか
- [ ] Content-Type, Cache-Control等の必要なヘッダーを返しているか

### セキュリティ
- [ ] HTTPS（TLS 1.3）を強制しているか
- [ ] HSTS ヘッダーを実装しているか
- [ ] API利用者認証（APIキー以上）を設けているか

### データ設計
- [ ] 日時はISO 8601形式（`YYYY-MM-DDTHH:mm:ss+09:00`）か
- [ ] 国コードはISO 3166-1 alpha-2か
- [ ] 都道府県コードはJIS X 0401か

## よくある質問への回答パターン

**Q: ページネーションのパラメータ名は？**
→ `limit`（件数）と `offset` または `page`（開始位置）を推奨。初期値は100件以下 [SHOULD]。10件以上返却する場合は `fields` パラメータも実装 [SHOULD]。

**Q: 非同期APIの完了をどう通知する？**
→ 受付時に `202 Accepted` + `Location` ヘッダー（状態確認URL）を返す。完了時は `303 See Other` + 結果URL。可能なら Webhook もサポートを検討 [MAY]。

**Q: エラーレスポンスのフォーマットは？**
→ RFC 7807準拠: `status`, `type`, `title`, `detail`, `instance` を含めることを推奨 [SHOULD]。Content-Typeは `application/problem+json`。
